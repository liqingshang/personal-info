## Context

The resume website already has a live homepage with five visual themes and a "Download PDF" button. The current PDF is generated by printing the homepage with `@media print` styles. While this works, it is not optimized for a standalone resume document: the layout is web-first, page breaks are uncontrolled, and the result looks like a printed web page rather than a designed PDF.

## Goals / Non-Goals

**Goals:**

- Create a dedicated `resume.html` page whose sole purpose is to produce a beautiful, print-ready PDF.
- Control page size (A4), margins, typography, page breaks, and widows/orphans.
- Provide a repeatable, one-command way to regenerate `assets/resume.pdf`.
- Update the homepage download button to serve the new PDF.
- Keep the PDF content consistent with the website.

**Non-Goals:**

- Changing the visual design of the public website or its themes.
- Adding a server-side PDF generator or paid API.
- Supporting multiple PDF layouts (one professional layout is enough).

## Decisions

### 1. Use a dedicated `resume.html` for PDF generation

- **Rationale**: Separating the PDF source from the interactive website lets us optimize typography and page breaks without affecting the site. It also avoids hiding interactive elements via print styles.
- **Alternative considered**: Keep improving the homepage `@media print` styles. Rejected because the homepage is designed for screens, not paper, and dark/ornate themes would produce inconsistent PDFs.

### 2. Print CSS uses `@page` and A4 sizing

- **Rationale**: `@page { size: A4; margin: 16mm; }` gives consistent output across Chrome's print-to-PDF. We will also use `page-break-inside: avoid` on sections and `widows/orphans` control.

### 3. Generate PDF with Chrome headless

- **Rationale**: Chrome is already installed in the development environment and produces reliable, vector PDFs from HTML. The command can be captured in a small Node script so it is repeatable.
- **Command**: `chrome --headless --disable-gpu --print-to-pdf="assets/resume.pdf" --run-all-compositor-stages-before-draw --virtual-time-budget=3000 "file:///.../resume.html"`

### 4. Content is copied from `index.html` initially, then kept in sync manually

- **Rationale**: There is no build system or templating engine. Duplicating the content into `resume.html` is the simplest path. A comment block in both files will mark the sections that must stay aligned.

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| Content drift between website and PDF | Add clear comments in both files and regenerate the PDF after any content change |
| Chrome-only generation command | Document the command; PDF is an artifact, so the command is only needed when content changes |
| Generating Chinese text in headless Chrome may have font issues | Use system fonts (`Noto Sans SC`, `PingFang SC`, `Microsoft YaHei`) in the print CSS |

## Migration Plan

1. Create `resume.html` with professional A4 print styles.
2. Copy current resume content from `index.html` into `resume.html`.
3. Create `scripts/generate-pdf.js` to wrap the Chrome headless command.
4. Run the script to produce `assets/resume.pdf`.
5. Update the "Download PDF" link in `index.html` to point to the same `assets/resume.pdf` (no path change needed).
6. Commit and push.

## Open Questions

- Should the PDF have a photo/avatar? (Current homepage has a placeholder.)
- Should the PDF include a QR code or link to the online version?
- Which color accent should the PDF use? (A neutral navy/amber is proposed to look professional when printed in grayscale or color.)
